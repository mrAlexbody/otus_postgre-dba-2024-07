
# Практические работу по курсу PostgreSQL DBA.

---
## Введение
Каждый модуль предполагает выполнение практических заданий, завершающихся коммитом с результатом.  
Для каждого модуля сформулированы цели, шаги выполнения и ожидаемые результаты.

## Модуль 01. Транзакции и уровни изоляции

**Цель:**  
Научиться работать с транзакциями, понимать различия уровней изоляции, выявлять и воспроизводить аномалии.

**Задания:**
1. Создайте тестовую таблицу `accounts(id int, balance numeric)` и наполните её данными.
2. В двух сессиях одновременно выполните:
   * `UPDATE` одной строки;
   * `UPDATE` разных строк одной таблицы;
   * вставку и удаление с разными уровнями изоляции (`READ COMMITTED`, `REPEATABLE READ`, `SERIALIZABLE`).
3. С помощью `pgbench` проведите нагрузочное тестирование сценария с конкурентным обновлением счётчиков.
4. Зафиксируйте в отчёте:
   * какие аномалии (`грязное чтение`, `неповторяемое чтение`, `фантомы`) наблюдаются на каждом уровне;
   * как ведёт себя `SERIALIZABLE` при конфликтах;
   * влияние параметра `default_transaction_isolation`.

**Ожидаемый результат:**  
Скрипты для воспроизведения аномалий, лог тестирования `pgbench`, краткий вывод о выборе уровня изоляции для высоконагруженных систем.

## Модуль 02. Docker

**Цель:**  
Освоить запуск PostgreSQL в контейнере для быстрого развёртывания тестовых сред.

**Задания:**
1. Напишите `Dockerfile` и `docker-compose.yml` для запуска PostgreSQL 16 с монтированием каталога для данных и настройками через переменные окружения.
2. Добавьте в compose‑файл второй сервис – `pgadmin` или `adminer` для визуального управления.
3. Подготовьте скрипт инициализации (`init.sql`), который создаёт тестовую БД и пользователя.
4. Проверьте:
   * сохранение данных после перезапуска контейнера;
   * возможность подключения извне (проброс портов);
   * ограничение ресурсов (CPU, память) для контейнера.

**Ожидаемый результат:**  
Рабочий `docker-compose.yml`, скрипты инициализации, краткая инструкция по запуску.

## Модуль 03. Установка и конфигурирование PostgreSQL

**Цель:**  
Научиться устанавливать PostgreSQL из разных источников (пакеты, исходники) и применять базовые настройки.

**Задания:**
1. Установите PostgreSQL:
   * из официального репозитория (apt/yum);
   * из исходного кода (опционально).
2. Настройте:
   * параметры `listen_addresses`, `port`, `max_connections`;
   * размещение `data_directory` на отдельном диске/разделе;
   * логирование (включите `log_connections`, `log_disconnections`, задайте ротацию логов).
3. Создайте кластер с нестандартным именем и портом.
4. Напишите systemd unit (если используется не из пакета) для управления службой.

**Ожидаемый результат:**  
Рабочий экземпляр PostgreSQL с пользовательскими настройками, конфигурационные файлы (`postgresql.conf`, `pg_hba.conf`), демонстрация запуска/остановки.

## Модуль 04. Работа с БД, пользователями и правами

**Цель:**  
Освоить управление ролями, привилегиями и схемами.

**Задания:**
1. Создайте роли:
   * `app_user` – для приложения (ограниченные права);
   * `admin_user` – с правами суперпользователя;
   * `readonly_user` – только чтение.
2. Создайте базу данных `testdb` и схему `app`.
3. Назначьте права:
   * `app_user` – полный доступ к схеме `app`;
   * `readonly_user` – доступ `SELECT` на все таблицы схемы `app`;
   * запретите `readonly_user` подключаться к другим базам.
4. Проверьте:
   * наследование привилегий через членство в ролях;
   * работу `ALTER DEFAULT PRIVILEGES` для новых таблиц.

**Ожидаемый результат:**  
Скрипт создания ролей и привилегий, демонстрация подключения с разными ролями и проверка ограничений.

## Модуль 05. Autovacuum

**Цель:**  
Настроить autovacuum для предотвращения раздувания таблиц и эффективного управления dead‑строками.

**Задания:**
1. Создайте таблицу с большим количеством `UPDATE`/`DELETE`.
2. Отключите autovacuum для этой таблицы и наблюдайте рост размера (`pg_relation_size`).
3. Включите autovacuum с агрессивными настройками (`autovacuum_vacuum_scale_factor = 0.01`, `autovacuum_vacuum_threshold = 50`).
4. Проанализируйте логи autovacuum (включив `log_autovacuum_min_duration`).
5. Напишите запросы для мониторинга:
   * последнего времени автовакуума;
   * количества мёртвых строк (`n_dead_tup` из `pg_stat_all_tables`);
   * раздутия таблиц.

**Ожидаемый результат:**  
Конфигурация параметров autovacuum, отчёт о поведении до и после настройки, скрипты мониторинга.

## Модуль 06. WAL (Write‑Ahead Log)

**Цель:**  
Изучить механизм WAL, настройку архивации и использование для восстановления.

**Задания:**
1. Настройте архивацию WAL в каталог `/wal_archive`.
2. Создайте базовую резервную копию (`pg_basebackup`).
3. Сымитируйте сбой:
   * удалите несколько файлов данных;
   * восстановите из бэкапа + подкатите WAL до последней транзакции.
4. Включите `wal_log_hints` и `full_page_writes`, проанализируйте влияние на размер WAL.
5. Определите, какие сегменты WAL нужны для PITR (восстановления на момент времени).

**Ожидаемый результат:**  
Настроенная архивация, сценарий восстановления с использованием `recovery.conf` (или `recovery.signal`), отчёт о размере WAL под нагрузкой.

## Модуль 07. Блокировки (Locking)

**Цель:**  
Научиться диагностировать и разрешать конфликты блокировок.

**Задания:**
1. В двух сессиях создайте ситуации:
   * блокировка строки (`UPDATE` без `COMMIT`);
   * блокировка таблицы (`LOCK TABLE`).
2. С помощью представления `pg_locks` найдите:
   * какие блокировки удерживаются;
   * кто и кого блокирует (waiting = true).
3. Настройте `deadlock_timeout` и `log_lock_waits` для выявления долгих ожиданий.
4. Сымитируйте deadlock и проверьте, как PostgreSQL его разрешает.

**Ожидаемый результат:**  
Скрипты для воспроизведения блокировок, методы мониторинга, настройки для логирования.

## Модуль 08. Нагрузочное тестирование и тюнинг

**Цель:**  
Провести нагрузочное тестирование и оптимизировать параметры PostgreSQL под конкретную нагрузку.

**Задания:**
1. С помощью `pgbench` инициализируйте тестовую БД с масштабом 100.
2. Запустите тест с разным количеством клиентов (10, 50, 100) и разными типами транзакций (скрипт по умолчанию, кастомный).
3. Соберите метрики: tps, latency, использование ресурсов (cpu, i/o).
4. Измените параметры:
   * `shared_buffers`, `effective_cache_size`, `work_mem`;
   * `wal_buffers`, `checkpoint_completion_target`;
   * `max_worker_processes`, `max_parallel_workers_per_gather`.
5. Сравните производительность до и после тюнинга, представьте графики/таблицы.

**Ожидаемый результат:**  
Отчёт с результатами тестов, обоснование выбора параметров для заданной нагрузки.

## Модуль 09. Резервное копирование

**Цель:**  
Освоить различные методы резервного копирования и восстановления.

**Задания:**
1. Выполните логический бэкап:
   * `pg_dump` отдельной базы;
   * `pg_dumpall` всех баз.
2. Выполните физический бэкап с помощью `pg_basebackup` с включённой архивацией WAL.
3. Восстановите из логического бэкапа в новую базу.
4. Настройте автоматическое создание бэкапов (cron или pg_cron) с ротацией.
5. Проведите восстановление до момента времени (PITR) с использованием архива WAL.

**Ожидаемый результат:**  
Скрипты бэкапов, демонстрация восстановления, план ротации.

## Модуль 10. Репликация

**Цель:**  
Настроить потоковую репликацию и освоить управление репликами.

**Задания:**
1. Разверните мастер и одну синхронную реплику (или асинхронную).
2. Убедитесь, что репликация работает:
   * `pg_stat_replication` на мастере;
   * `pg_is_in_recovery()` на реплике.
3. Настройте каскадную репликацию (вторая реплика подключается к первой).
4. Выполните переключение (switchover) и отработку отказа (failover) с помощью `pg_ctl promote` или `pg_promote`.
5. Организуйте мониторинг лага репликации (`pg_last_wal_receive_lsn` и `pg_last_wal_replay_lsn`).

**Ожидаемый результат:**  
Рабочий кластер репликации, документация по процедурам переключения, скрипты мониторинга.

## Модуль 11. Индексы

**Цель:**  
Научиться выбирать и создавать эффективные индексы для различных типов запросов.

**Задания:**
1. Создайте таблицу с 1 млн строк и несколькими столбцами (включая текстовые и JSON).
2. Сгенерируйте несколько типовых запросов:
   * поиск по равенству;
   * диапазонный поиск;
   * полнотекстовый поиск;
   * поиск по выражению.
3. Подберите подходящие типы индексов (B‑tree, GIN, GiST, BRIN) и сравните план выполнения до и после создания.
4. Используйте покрывающие индексы (INCLUDE) для ускорения `SELECT` без обращения к таблице.
5. Проведите нагрузочное тестирование, чтобы оценить влияние индексов на вставки/обновления.

**Ожидаемый результат:**  
Скрипты создания индексов, анализ `EXPLAIN (ANALYZE, BUFFERS)`, вывод о выборе индекса.

## Модуль 12. Статистика и JOIN‑оптимизация

**Цель:**  
Изучить работу планировщика, собирать и анализировать статистику, оптимизировать соединения.

**Задания:**
1. Создайте две таблицы, связанные по ключу, с неравномерным распределением данных.
2. Проверьте планы для разных типов JOIN (`Nested Loop`, `Hash Join`, `Merge Join`).
3. Измените параметры `join_collapse_limit`, `from_collapse_limit`, `enable_*` и проследите за изменением плана.
4. Выполните `ANALYZE` и проанализируйте содержимое `pg_stats`.
5. Сымитируйте ситуацию, когда статистика устарела, и продемонстрируйте, как это влияет на выбор плана.

**Ожидаемый результат:**  
Анализ планов запросов, настройка сбора статистики, рекомендации по управлению `default_statistics_target`.

## Модуль 13. Партиционирование таблиц

**Цель:**  
Внедрить декларативное партиционирование для больших таблиц.

**Задания:**
1. Создайте таблицу с партиционированием по диапазону (например, по дате) или по списку.
2. Наполните данными несколько партиций.
3. Продемонстрируйте:
   * отсечение партиций (partition pruning) в запросах;

4. Добавьте партицию с включённым `FOREIGN KEY` и проверьте ограничения.
5. Выполните реорганизацию: прикрепите существующую таблицу как партицию, используя `ATTACH PARTITION`.

**Ожидаемый результат:**  
Скрипты создания партиционированной таблицы, примеры запросов с `EXPLAIN`, демонстрация управления партициями.

## Модуль 14. Триггеры

**Цель:**  
Написать триггеры для автоматизации бизнес‑логики и аудита.

**Задания:**
1. Создайте таблицу аудита `audit_log` (столбцы: operation, old_data, new_data, changed_at, user_name).
2. Напишите функцию‑триггер для логирования всех изменений в целевой таблице (`INSERT`, `UPDATE`, `DELETE`).
3. Добавьте событийный триггер для логирования DDL (`CREATE TABLE`, `ALTER TABLE`).
4. Создайте триггер, который автоматически обновляет поле `updated_at` при изменении строки.
5. Проверьте производительность: как много триггеры добавляют к времени выполнения DML.

**Ожидаемый результат:**  
Код триггеров и функций, демонстрация работы, замеры производительности.

## Итоговый проект (Project04)

**Цель:**  
Комплексное задание, объединяющее все изученные темы.

**Предполагаемые задачи:**
1. Развернуть отказоустойчивый кластер PostgreSQL (мастер + синхронная репликация) в Docker или на виртуальных машинах.
2. Настроить:
   * автовакуум с агрессивными параметрами для нескольких таблиц;
   * архивацию WAL и регулярные бэкапы (pg_basebackup + архив WAL);
   * мониторинг (например, через pg_stat_statements и системные представления);
3. Создать партиционированную таблицу для хранения логов (по дате) и триггер аудита для неё.
4. Провести нагрузочное тестирование (pgbench) и оттюнить параметры PostgreSQL для достижения целевого TPS.
5. Продемонстрировать процедуру аварийного переключения (failover) и восстановление после сбоя мастера с использованием реплики и бэкапов.
6. Подготовить отчёт: конфигурации, скрипты, графики, объяснение принятых решений.

**Ожидаемый результат:**  
Полная документация проекта, скрипты развёртывания и управления, демонстрация работоспособности кластера и процедур восстановления.
